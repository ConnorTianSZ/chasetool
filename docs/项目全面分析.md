## 一、项目概述

ChaseBase 是博世（Bosch）BMG-XCN/PUL 部门内部使用的**采购催货管理系统**，帮助采购团队追踪物料交货期、发送催货邮件、处理供应商回复邮件、管理交货风险仪表板。

### 核心目标

- 从 SAP 导出采购订单（PO）Excel 文件，导入系统
- 根据物料状态（逾期、临近关键节点、无确认交期等）自动分组生成催货邮件
- 通过 Outlook 集成发送/接收邮件，追踪供应商回复
- 利用 LLM（Claude / DeepSeek）智能解析供应商回复邮件，提取交期信息
- Dashboard 可视化展示风险，支持按采购员、供应商、制造商等多维度分析
- 项目关键节点（Key Date / Time Nodes）管理，对比交期是否满足项目需求

## 二、技术架构

### 2.1 整体架构

```
[浏览器 (Alpine.js SPA)] ←→ [FastAPI 后端] ←→ [SQLite 数据库（每项目独立）]
                               ↕                    ↕
                         [Microsoft Outlook]    [LLM API]
                         (win32com COM)     (Claude/DeepSeek)
```

### 2.2 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 后端框架 | FastAPI (Python 3.13) | 异步 Web 框架 |
| 前端 | Alpine.js 3 + 原生 JS + Chart.js | 轻量 SPA |
| 数据库 | SQLite 3 | 每项目一个独立 .db 文件 |
| 邮件集成 | pywin32 (win32com) | Windows Outlook COM 自动化 |
| 邮件解析 | extract-msg | .msg 文件解析 |
| Excel 处理 | openpyxl + pandas | 导入/导出 SAP Excel |
| LLM | Anthropic Claude / DeepSeek (OpenAI 兼容) | 聊天、邮件解析、催货内容生成 |
| 配置管理 | pydantic-settings + YAML | .env + 映射配置 |
| 样式 | 原生 CSS + CSS 变量 | 无前端框架依赖 |

## 三、核心功能模块

### 模块概览

```
项目管理 ─── 创建/切换/删除项目，每项目独立数据库
物料管理 ─── 列表/筛选/状态/编辑/删除/重点标记
Excel导入 ─── 解析SAP导出格式，WBS解析，PGR映射，增量更新
催货邮件 ─── 自动状态推断，分组生成，Outlook草稿/发送，邮件追踪标记
收件箱 ─── Outlook收件箱扫描，LLM解析供应商回复，决策执行
Dashboard ─── 概览/聚合/采购员视图/时间节点/透视表
LLM聊天 ─── 自然语言查询和操作
```

### 3.1 项目管理（`app/api/projects.py`）

- **功能**: 多项目支持，每项目一个独立 SQLite 数据库
- **注册表**: `data/projects.json` 记录所有项目元信息
- **数据隔离**: 每个项目的数据库文件位于 `data/{project_id}/chasebase.db`

### 3.2 物料管理（`app/api/materials.py`）

- **核心 CRUD**: 列表、筛选、详情、更新、删除、历史记录
- **丰富筛选**: 支持按 PO号、采购员、供应商、状态、物料状态、站点、采购组、重点标记、搜索关键词、逾期、无交期、已催未回复等条件组合筛选
- **物料状态实时推导**: 每次列表查询时，根据 current_eta、urgent_feedback_eta、open_quantity_gr、key_date 等计算物料状态（详见 6.1 节）
- **重点标记**: 标记/取消重点物料，界面高亮显示

### 3.3 Excel 导入/导出（`app/services/excel_io.py`）

- **导入**:
  - 解析 SAP 导出的标准采购报表 .xlsx 文件
  - 通过 YAML 映射配置（`config/excel_column_mapping.yaml`）自动匹配列名（支持中英文别名）
  - 自动解析 WBS 元素，提取项目号（project_no）和工位号（station_no）
  - 根据采购组（PGR）代码自动填充采购员姓名和邮箱
  - 去重逻辑：以 (po_number, item_no) 为联合键，已有记录则更新，新增记录则插入
  - 数量为 0 的行自动跳过
  - 记录导入日志（imports 表）
  - 字段更新通过 `update_policy.py` 治理（详见 6.3 节）

- **导出**:
  - `export_full_db()`: 全量导出为 Excel，重点物料黄色突出显示
  - `export_back()`: 将 current_eta 和 status 写回源 Excel 副本
  - `export_chase_append()`: 在源 Excel 末尾追加催货信息列

### 3.4 催货邮件（`app/api/chase.py` + `app/tools/chase_email.py`）

- **自动状态推断**: 根据每条物料的 `derive_material_state()` 结果决定催货类型
  - `no_oc`（无OC）→ `oc_confirmation` 模板（确认交期）
  - `overdue_now`（应交未交）→ `urgent_now` 模板（加急催交期）
  - `overdue_keydate`（晚于节点）→ `urgent_keydate` 模板（请求提前）
  - `delivered` / `normal` → 自动跳过

- **分组发送**: 按 `(buyer_email, project_no, chase_type)` 分组，跨 PO 合并为一封邮件

- **邮件模板管理**: 存放在 `config/chase_email_templates/` 目录：
  - `oc_confirmation.txt` — 确认 OC 交期（HTML 模板）
  - `urgent_now.txt` — 当前已逾期（加急）
  - `urgent_keydate.txt` — 晚于关键节点（请求提前）
  - `urgent.txt` — 通用加急（遗留）
  - `default.txt` — 纯文本 fallback

- **邮件追踪标记**: 每封邮件 subject 中嵌入 `[CB:...]` 标记（详见 6.4 节）

- **发送模式**: `draft`（保存草稿到 Outlook）/ `send`（直接发送）

### 3.5 收件箱处理（`app/api/inbox.py` + `app/services/outlook_inbox.py` + `app/tools/parse_inbound.py`）

- **拉取邮件**: 通过 Outlook COM 扫描收件箱，筛选 subject 带 `[CB:...]` 标记的邮件
- **回溯天数**: 默认从最近一次发送催货邮件的时间回溯，最多深拉 90 天
- **去重**: 通过 `outlook_entry_id` 唯一键防止重复入库
- **LLM 解析**: 调用 LLM 解析供应商回复邮件，提取每个 PO+Item 的 `new_eta` 和 `remarks`
  - 支持中文非正式表达，如"今天"、"明天"、"月底"等相对日期
  - 支持一条回复申明覆盖所有物料的场景
  - 返回结构化 JSON：`{items: [{po_number, item_no, new_eta, remarks}], general_remarks, confidence}`
- **决策应用**:
  - `apply` — 将提取的交期写入 `supplier_eta`、`supplier_remarks`，记录反馈时间
  - `ignore` — 忽略此邮件
  - `manual` — 标记为需要人工处理
  - 分批次处理，支持部分应用（partial status）
- **拒绝回复**: 对供应商回复回复"不接受"邮件，通过 Outlook `ReplyAll` 发送，要求供应商提前交期

### 3.6 Dashboard（`app/api/dashboard.py`）

- **概览**: 总物料数、进行中、逾期数、无交期、重点数、已交货
- **聚合统计**: 按状态/供应商/采购员/站点/采购组分组计数
- **按供应商逾期排名**
- **催货统计**: 采购员维度统计：总物料、总催货次数、已回复数
- **采购员视图（Lead Buyer View）**:
  - 每个采购员的风险物料汇总（无OC、应交未交、晚于节点、已催未回复、重点数）
  - 支持 `current_eta` / `supplier_eta` 切换
  - 顶部制造商维度汇总
  - 导出为 Outlook HTML 草稿邮件
- **时间节点管理（Time Nodes）**:
  - 创建/编辑/删除关键项目节点（里程碑日期）
  - 每个节点统计到期物料数、无交期数
  - 按采购员/供应商钻取到期物料明细
- **透视表**:
  - `pivot_buyer_docdate`: 采购员 × 下单日期 交叉统计（no_oc / overdue_now / overdue_keydate）
  - `pivot_buyer_manufacturer`: 采购员 × 制造商 交叉统计（晚于节点）

### 3.7 LLM 聊天（`app/api/chat.py` + `app/tools/registry.py`）

- **自然语言交互**: 采购员可通过聊天界面用中文向系统提问或下发指令
- **可用工具（注册在 `app/tools/`）**:
  - `get_overview` — 获取项目概览
  - `query_aggregates` — 分组聚合统计
  - `search_materials` — 多条件搜索物料
  - `get_material` — 获取单个物料详情
  - `update_material_field` — 更新物料字段（`supplier_eta`, `supplier_remarks`, `status`, `is_focus`，**不允许**修改 `current_eta`）
  - `mark_focus` — 切换重点标记
  - `generate_chase_drafts` — 生成催货草稿
  - `send_chase_drafts` — 发送催货草稿
  - `parse_inbound_email` — 解析供应商回复邮件
  - `apply_inbound_decision` — 应用邮件处理决策
- **系统提示词**: 自定义身份为"BMG-XCN/PUL ChaseBase 采购 AI 助手"，不透露底层模型

## 六、核心业务逻辑详解

### 6.1 物料状态推导（`derive_material_state()`）

系统定义了 **6 种物料状态**，每次列表查询时实时推导：

```
                                     ┌──────────────────┐
                                     │  open_quantity_gr │
                                     │  = 0 ?           │
                                     └───────┬──────────┘
                                             │ Yes
                                             ▼
                                     ┌──────────────┐
                                     │  已交货       │
                                     │  (delivered)  │
                                     └──────────────┘
                                             │ No
                                             ▼
                                     ┌──────────────────┐
                                     │  current_eta     │
                                     │  为空?           │
                                     └───────┬──────────┘
                                             │ Yes
                                             ▼
                                     ┌──────────────┐
                                     │  无OC         │
                                     │  (no_oc)      │
                                     └──────────────┘
                                             │ No
                                             ▼
                    ┌─────────────────────────────────────────┐
                    │ current_eta < today OR                   │
                    │ current_eta > key_date ?                 │
                    │ (即 current_eta 需要加急)                 │
                    └───────────────────┬─────────────────────┘
                                        │ No (正常范围内)
                                        ▼
                                ┌──────────────┐
                                │  正常         │
                                │  (normal)    │
                                └──────────────┘
                                        │ Yes
                                        ▼
                    ┌─────────────────────────────────────────┐
                    │  urgent_feedback_eta 存在?               │
                    │  (采购员手工记录了催后确认交期)             │
                    └───────────────────┬─────────────────────┘
                                        │ No
                                        ▼
               ┌────────────────────────────────────────────────┐
               │  current_eta < today ?                         │
               │  → overdue_now (应交未交)                      │
               │  current_eta > key_date ?                      │
               │  → overdue_keydate (晚于节点)                  │
               └────────────────────────────────────────────────┘
                                        │ Yes (urgent_feedback_eta 存在)
                                        ▼
               ┌────────────────────────────────────────────────┐
               │  urgent_feedback_eta 是否在可接受范围内?         │
               │  (>= today 且 <= key_date)                     │
               ├────────────────────┬───────────────────────────┤
               │ ✅ 可接受           │ ❌ 不可接受                │
               │ → eta_mismatch     │ → 同无 urgent 时逻辑      │
               │   (交期与反馈不一致) │   overdue_now/overdue    │
               └────────────────────┴───────────────────────────┘
```

**状态汇总表**:

| 状态码 | 中文标签 | 说明 |
|--------|----------|------|
| `delivered` | 已交货 | `open_quantity_gr = 0` |
| `no_oc` | 无OC | 未交货且 `current_eta` 为空 |
| `overdue_now` | 应交未交 | `current_eta < today`（已过原定期限） |
| `overdue_keydate` | 晚于节点 | `current_eta > key_date`（无法满足项目节点） |
| `eta_mismatch` | 交期与反馈不一致 | SAP 交期异常但采购员确认催后新交期可接受 |
| `normal` | 正常 | `today <= current_eta <= key_date` |

### 6.2 催货状态推导（`derive_chase_status()`）

| 状态码 | 中文标签 | 说明 |
|--------|----------|------|
| `feedback` | 已于 X 月 X 日第 N 次反馈 | 有 `supplier_feedback_time` |
| `chased_no_feedback` | 已第 N 次于 X 月 X 日催未反馈 | `chase_count > 0` 但无反馈 |
| `not_chased` | 未催 | `chase_count = 0` |

### 6.3 字段更新优先级（`update_policy.py`）

多源写入冲突治理，按优先级决定操作是否执行：

| 优先级 | 来源 | 说明 |
|--------|------|------|
| **4** | `buyer_manual` | 采购员手工录入（最高，不被任何覆盖） |
| **3** | `email_reply` | 供应商邮件回复 |
| **2** | `chat_command` | LLM 聊天指令 |
| **1** | `manual_import` | Excel 导入 |
| **0** | `system` | 系统内部 |

**敏感字段**（`SENSITIVE_FIELDS`）: `current_eta`, `supplier_eta`, `supplier_remarks`, `status`, `is_focus`, `urgent_feedback_eta`, `urgent_feedback_note` — 低优先级来源不能覆盖高优先级来源已写入的值。

**特别规则**:
- `current_eta` **只允许** `manual_import`（Excel 导入）修改，chat 和 email 均被拦截
- `urgent_feedback_eta` / `urgent_feedback_note` 只允许 `buyer_manual` 写入，**不被 Excel 导入覆盖**
- 事实字段（`FACT_FIELDS`）如 `po_number`, `part_no`, `supplier` 等不受优先级保护，Excel 导入可直接更新

### 6.4 邮件追踪标记协议（`email_marker.py`）

在每封催货邮件的 subject 中嵌入 `[CB:...]` 标记，用于接收供应商回复时关联原始催货：

**v2 格式（当前使用）**:

```
[CB:{project_no}-{pgr}-{purpose}-{MMDD}-{seq}] {标题}
```

| 部分 | 含义 | 示例 |
|------|------|------|
| `project_no` | 项目号 | `M.6001515`、`P2024001` |
| `pgr` | 采购组代码 | `MFW`、`MFB` |
| `purpose` | 用途 | `OC`（确认交期）/ `URG`（加急） |
| `MMDD` | 发送日期 | `0507` |
| `seq` | 当日同 base 序号 | `1`、`2` |

示例: `[CB:M.6001515-MFW-OC-0507-1] OC Confirmation / 请确认订单交期`

**v1 旧格式（兼容解析）**:
```
[CB:{PO}/{ITEM_NOS}]
```

**兼容处理**:
- 供应商邮件客户端有时将 `[CB:` 变为 `[CB.`，解析时自动标准化
- 项目号允许包含点号（如 `M.6001515`）

### 6.5 Excel 列名映射（`excel_column_mapping.yaml`）

SAP 导出的 Excel 列头可能包含英文/中文/缩写等多种变体，映射文件配置了每个数据库字段对应的所有可能列头名。匹配规则：trim + 大小写不敏感 + 全半角统一（NFKC normalization）。未匹配的列自动存入 `extra_json`。

### 6.6 采购组映射（`purchasing_group_mapping.yaml`）

将 SAP 采购组代码（如 `MFW`、`MFB`）映射为采购员姓名和邮箱。Excel 导入时自动按 PGR 填充采购员信息。

### 6.7 Key Date（关键节点日期）

每个项目可设置一个 Key Date，用于物料状态推导。`current_eta > key_date` 的物料被判定为"晚于节点"。Key Date 通过 API 可动态调整。

### 6.8 仪表板 eta_source 切换

Dashboard 支持在 `current_eta` 和 `supplier_eta` 之间切换显示。当选择 `supplier_eta` 时，用供应商反馈交期替换当前交期作为判断依据（空则 fallback 为 current_eta）。
